## Confirmed but not fixed
- BUG-001 · high · security · flask_security/core.py:1217 · is_locked() docstring/contract says "return True if locked" but body returns True and every call site uses `if not user.is_locked(): return False`, so any override following the documented contract inverts lockout (locked accounts admitted, valid accounts blocked)
- BUG-002 · high · security · flask_security/webauthn.py:332 · is_active check appends DISABLED_ACCOUNT error but is missing `return False` (unlike forms.py:463-465), so a deactivated account with a passkey falls through and completes WebAuthn sign-in
- BUG-003 · high · data-integrity · flask_security/recovery_codes.py:105 · idx is computed from the decrypted+TTL-filtered code list but mf_delete_recovery_code pops that index from the stored encrypted list, so when MULTI_FACTOR_RECOVERY_CODE_TTL is set and an earlier code expired, the wrong code is deleted and the just-used recovery code stays replayable
- BUG-004 · medium · data-integrity · flask_security/mail_util.py:140 · config_value() returns the live app.config dict, so `validator_args["check_deliverability"]=False` permanently mutates SECURITY_EMAIL_VALIDATOR_ARGS, silently disabling deliverability checks for all later validate() calls
- BUG-005 · low · error-handling · flask_security/oauth_provider.py:113 · GitHub fetch_identity_cb returns profile["email"] directly, which is None for private/unverified emails (→ find_user(email=None)) and raises KeyError if the key is absent
- BUG-006 · low · data-integrity · flask_security/utils.py:781 · redirect userinfo is rebuilt as `f"{u.username}:{u.password}@"` even when only a username is present, injecting a literal `:None` password and corrupting an otherwise valid redirect URL
- BUG-007 · low · error-handling · flask_security/recovery_codes.py:105 · delete_recovery_code uses `codes.index(code)`, which raises an uncaught ValueError (HTTP 500) if the code was already consumed by a concurrent submit
