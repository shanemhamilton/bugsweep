The one thing to understand

The worst case is a branch you delete.

During the hunt-and-fix loop, bugsweep never touches your real branch, never pushes, never merges, and never deletes a file. It cuts a fresh bugsweep/<timestamp> branch, lands one commit per bug, and re-runs your tests after every fix — automatically reverting any fix that breaks something. You are always the merge gate.

The irreversible git operations are done by short, auditable shell scripts you can read in a few minutes — not left to the model's judgment. That's what makes it trustworthy to leave running overnight.

Why the false-positive rate stays low

Every finding faces a tribunal.

Most AI reviewers grade their own homework. bugsweep doesn't. A candidate bug has to survive three independent roles before it's ever fixed or even reported — the model never rubber-stamps its own guess.

01
Finds

The Hunter

Sweeps a batch of files against a whole-repo architecture model and a stack-specific anti-pattern library, and raises candidate bugs with the exact evidence trail.

02
Disproves

The Skeptic

Actively tries to kill each candidate — alternate explanations, guarding code paths, tests that already catch it — tuned to punish dismissing a real bug more than missing a false alarm.

03
Decides

The Referee

A neutral arbiter reads the code independently and issues the final verdict. Only its confirmed list is eligible to be fixed — and it seeds variant queries to hunt siblings next time.

Hunter → Skeptic → Referee — confirmed bugs go to the fix queue; everything else is dropped, not reported.

The moat: whole-repo context

It catches the bugs a single prompt can't see.

Before hunting, bugsweep distills your architecture — trust boundaries, sensitive sinks, and the call chains into them. That's how it finds cross-file bugs: a missing authorization check on one path into a database write, contract drift across a module boundary, untrusted input reaching a sink three files away.

A real find, not a lint warning

In a benchmark case whose planted bug was a cross-package SSRF chain, a single-prompt baseline review missed it entirely. bugsweep's architecture model traced untrusted input from one package into an outbound request in another — and flagged it.

It ignores the noise on purpose. Style, formatting, naming, unused imports, TODOs, dependency versions — those are a linter's job. bugsweep spends its attention on bugs that actually fault at runtime.

Coverage-first · not a diff scanner

It gets deeper every night.

The whole repo is always in scope — bugsweep finds latent bugs in old, unchanged code, not just your last diff. It keeps a map on disk of what's been audited and reprioritizes toward the never-seen, the stale, and the high-risk. The repo is never permanently "done" while a frontier remains.

audited & fresh stale / high-risk never seen
  • Night 1Builds the architecture model, sweeps the highest-risk sinks, lands the first fixes.
  • Night 2Skips what's fresh, pushes the frontier into never-audited modules.
  • Night 3Replays variant queries from earlier finds — hunts the whole repo for siblings of confirmed bugs.
  • OngoingRe-checks stale files as the catalog evolves. Survives context resets — all progress lives on disk.
Built for the night shift

Point it at a repo. Read the report at breakfast.

bugsweep is designed to run unattended on a schedule — and to end every session in a clean git state, with no pile of leftover branches.

Runs headless

Drive it from claude -p under cron or launchd. It auto-triggers from a plain-English task — no interactive slash command needed.

Won't collide

An activity-aware pre-check defers politely if another session is working the tree, and closes stale work without ever discarding it.

Bounded & capped

Iteration, runtime, and fix caps keep cost predictable. It stops when it converges and writes a report either way.

Auditable

Every iteration is appended to a run ledger on disk. Findings are emitted as a machine-readable block for CI and dashboards.
