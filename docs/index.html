The one thing to understand

The worst case is a branch you delete.

During the hunt-and-fix loop, bugsweep never touches your real branch, never pushes, never merges, and never deletes a file. It cuts a fresh bugsweep/<timestamp> branch, lands one commit per bug, and re-runs your checks after every fix — automatically reverting any fix that breaks something. You are always the merge gate.

The irreversible git operations are done by short, auditable shell scripts you can read in a few minutes — not left to the model's judgment. That's the whole trust contract, and it's what makes it safe to leave running overnight: nine non-negotiable rules, enforced by scripts, not prompts.

01

Throwaway branch only. bugsweep/<timestamp> — never your branch.

02

No remotes, ever, during the run. No push, pull, fetch, PR, or merge. You are the only merge gate.

03

No destructive operations. No force, no reset --hard on your content, no deletes, no rewritten history.

04

Your work is preserved. Preflight stashes, finalize restores — your branch and tree end exactly as they began.

05

One bug, one commit, auto-revert. Checks re-run after every fix; a regression gets reverted.

06

Only confirmed bugs get fixed. Every finding survives adversarial review first.

07

Minimal, surgical fixes only. No refactors, renames, or reformatting.

08

Hard caps. Iterations, runtime, and fix count are bounded — it stops when it converges.

09

Everything is logged. Every iteration lands in a run ledger — an overnight run is auditable.

Why the false-positive rate stays low

Every finding faces a tribunal.

Most AI reviewers grade their own homework. bugsweep doesn't. A candidate bug has to survive three independent roles before it's ever fixed or even reported — the model never rubber-stamps its own guess.

01
Finds

The Hunter

Sweeps a batch of files against a whole-repo architecture model and a stack-specific anti-pattern catalog, and raises candidate bugs with the exact evidence trail.

02
Disproves

The Skeptic

Actively tries to kill each candidate — alternate explanations, guarding code paths, tests that already catch it — tuned to punish dismissing a real bug more than missing a false alarm.

03
Decides

The Referee

A neutral arbiter reads the code independently and issues the final verdict. Only its confirmed list is eligible to be fixed — and it seeds variant queries to hunt siblings next time.

Hunter → Skeptic → Referee — confirmed bugs go to the fix queue; everything else is dropped, not reported.

Built for a fleet, not one session

The overnight orchestrator.

A loop that runs unattended isn't enough — a scheduler needs to trust what it kicked off. bugsweep ships the plumbing: a machine-readable result contract, isolation for running many hunts on one repo at once, and teardown that never guesses.

A contract a scheduler can read

Every run reduces its ledger into a schema-valid run-summary.json — status, coverage, severity counts, fixed/quarantined/confirmed-unfixed findings, and where to look next. It always exists after finalize; if full reduction can't run, a minimal summary ships instead, flagged degraded: true.

Multiple subagents, one repo

Preflight's --worktree mode gives each run its own linked git worktree, so a fleet of isolated subagents can hunt the same repository concurrently without colliding. Your working branch, index, and files are never touched.

Always finalizes

A deadline checkpoint runs at every phase boundary. When the clock runs out, the run finalizes anyway — branch restored, summary written — instead of dying mid-hunt with nothing on disk.

Crash-safe teardown

An optional reaper reclaims worktrees and branches only on positive evidence a run is dead and done. Anything dirty, unmerged, or ambiguous is preserved — nothing unreviewed gets deleted.

Merges are re-verified, one at a time

integrate.sh lands multiple confirmed fix branches into a target branch serially, re-running proof after each merge — outside the core loop, and always gated by a human.

Reverts, but understands "flaky"

A newly-failing test is rerun up to 3 times before a fix is reverted; only a strict majority of passing reruns reclassifies it FLAKY. Reruns share the run's own tree, so this isn't full isolation — any fix landing as flaky is always surfaced, never silent.

Analyzer corroboration, opt-in

Turn on .analyzers.enabled and installed static analyzers — semgrep, gosec, bandit, and others already on your machine — seed the hunt as an independent signal. Off by default.

The moat: whole-repo context

It sees the bugs a single prompt can't.

Before it hunts, bugsweep distills your architecture — trust boundaries, sensitive sinks, and the call chains that reach them — and primes itself with a curated, stack-specific anti-pattern catalog (JS/Node, TypeScript, React, Python, Go, Kotlin, Swift/iOS, and more). That's how it catches cross-file bugs: a missing authorization check three hops from the sink, contract drift across a module boundary, untrusted input that only turns dangerous two files downstream.

run-summary.json — the machine-readable contract 
// schemas/run-summary.schema.json (shape, not sample data)
{
  "status": "complete" | "partial" | "stalled",
  "coverage": { audited, frontier, stale … },
  "counts": { critical, high, medium … },
  "findings": [ { evidence trail per bug } ],
  "fixed" / "quarantined" / "confirmed_unfixed": […],
  "root_cause_clusters": […],
  "follow_up": "where to look next"
}

✔ always written by finalize.sh — degraded:true if the
  full reducer can't run, never silently absent

No prose for a scheduler to parse

A scheduler — cron, launchd, a CI job, a nightshift dispatcher — can branch on run-summary.json without an LLM in the loop. schemas/run-summary.schema.json is the contract; scripts/summarize.sh is what fills it in from the run ledger.

It ignores the noise on purpose, too. Style, formatting, naming, unused imports, dependency versions — that's a linter's job. bugsweep spends its attention on bugs that actually fault at runtime.

Coverage-first · not a diff scanner

It gets deeper every night.

The whole repo is always in scope. .bugsweep/state/ persists an audit log, a risk map, and the anti-pattern catalog version across runs, and each run reprioritizes toward the never-audited, the stale, the content-changed, and the high-risk. The repo is never permanently "done" while a frontier remains.

audited & fresh stale / high-risk never seen
  • Night 1Builds the architecture model, sweeps the highest-risk sinks, lands the first fixes.
  • Night 2Skips what's fresh, pushes the frontier into never-audited modules.
  • Night 3Replays variant queries from earlier finds — hunts the whole repo for siblings of confirmed bugs.
  • OngoingRe-checks stale files as the anti-pattern catalog gains detector classes. Survives context resets — all progress lives on disk, aggregated across runs.
Benchmark harness · pilot evidence

A harness built to be hard to fool.

bench/ runs bugsweep headless against throwaway clones of dated, real CVE-class bugs pinned to a pre-fix commit, scores the output through an independent judge, and refuses to round a small sample up into a rate.

  • ▸Hardened container — non-root UID 65534, --read-only root + tmpfs, --cap-drop=ALL, no-new-privileges, CPU/memory/PID caps.
  • ▸Pinned pre-fix commit — each case clones a dated repo at a known-vulnerable SHA and fails closed if HEAD drifts.
  • ▸One network path — an nginx egress reverse proxy to a single hardcoded upstream (api.anthropic.com) on an --internal Docker network. Nothing else is reachable.
  • ▸Two-stage scoring — a hard file-overlap gate, then a cross-model LLM judge (a different vendor's model grading Claude's findings), with optional human calibration.
  • ▸Reported with rigor — detection rate with a 95% Wilson confidence interval, a paired delta against a no-skill claude -p baseline, and a contamination split by disclosure date.
  • ▸Corpus — a 9-case CVE-class corpus across Go, JavaScript, Python, and TypeScript.
Read this before the numbers below

Every leaderboard the harness has produced is labeled inconclusive. Reporting a rate requires at least 6 completed post-cutoff cases; that floor has never been reached — most cases still error or skip inside the locked-down container. There is no aggregate detection rate here. What follows are directional, single-case (k=1) pilot results, not a statistically powered rate.

Get started

One command. No dependencies.

Installs as a skill for Claude Code and Codex — auto-detects ~/.claude and ~/.codex. Then just ask it to find bugs, or run /bugsweep --autonomous for the overnight hunt.

$ curl -fsSL https://raw.githubusercontent.com/shanemhamilton/bugsweep/main/install.sh | bash
/bugsweep detect-only  ·  --fix confirm & patch  ·  --approve ask first  ·  --autonomous loop all night

--claude / --codex / --all to target an install  ·  --version vX.Y.Z to pin — see Releases for tagged versions.
